# ゲーム完成のためのテスト計画 — ループ周回と配線監査でギャップをあぶり出す

> 本プロジェクトは **Core の純ロジックが巨大で、その多くが「盤面に未配線／効かない」**のが完成の最大リスク（CLAUDE.md の観測層節「Core は増えるが見えない/効かない乖離」）。
> 完成のためのテストは **各ループが閉じるか＋効果がプレイヤーに届くか** を周回検証し、足りない配線をあぶり出す形が最も効く。
> 各テストの型：**「Xが一周するかをテスト観点に、足りない機能をあぶり出す。配線はその後で承認の上、最小で」**。検証は EditMode/TestHarness で CI 可能なテストに落とし、ギャップは doc に列挙する。

---

## 0. 完成の定義（縦スライス）

完成＝**タイトル → 戦略マップで1キャンペーンを通しプレイ → 会戦に潜行して勝つ → セーブ/ロードして続けられる**。この最小経路を壊さないことが全テストの上位ゴール。

**まず安全網を回す**
> EditMode 全テストと TestHarness（dotnet）を回し、赤を一覧化して原因と修正方針を出す。さらに `-ginei-playtest` で AI 対 AI 会戦を回し、例外/未決着/全停止/Material リークの所見を報告する。

## 横断の型（毎回つける一文）

> **Core 純ロジックは追加せず既存窓口を呼ぶ**こと。検証は EditMode/TestHarness で CI 可能なテストに落とし、足りない配線は doc にギャップ列挙してから、承認の上で最小配線する。

**推奨着手順**：`L5（セーブ）→ L1/L2/L3（各ループ）→ C1（orphan監査）→ C2-C6（因果）→ P1（操作到達）→ S1（ラグ）`。
理由：壊れると致命的な順（保存・主要ループ）→ 効かない腐り（配線）→ 届かない機能（操作）→ 品質ゲート（ラグ/決定論）。
