# 人月の神話 参考設計（EPIC #MMM）

> 参照元：『人月の神話』（フレデリック・P・ブルックス／The Mythical Man-Month）。巨大ソフトウェア開発（OS/360）の現場から導かれた**工数とスケジュールの非線形法則**の古典。
> 中核命題：**「遅れているプロジェクトに人を足すとさらに遅れる」（ブルックスの法則）／「銀の弾丸はない」**。人月は人と月が交換可能だという神話への反証であり、調整コスト・概念の完全性・段取りの本質的困難を説く。
> 本ドキュメントは、当プロジェクト（Ginei＝銀英伝風の星間国家戦略＋造船#884/メガプロジェクトPIL/省庁MinistryRules 等の生産・組織純ロジック層）にとって**役に立つ視点**だけを抽出し、EPIC `#MMM` として issue 化する提案。
> 著作権注意：固有名・文章・図表・固有設定は流用せず、**工数法則／組織構造のパターンのみ**を参考にする。

## 0. なぜ「人月の神話」が本システムに役立つか

当プロジェクトは生産・建艦・組織編制の**純ロジックを大量に保有**している（[CLAUDE.md]「テスト基盤」節 参照）：

| 既存（生産・組織の土台） | カバー範囲 |
|---|---|
| `ShipyardRules`/`Shipyard`/`BuildOrder`（#884） | 建艦キュー・`buildPower`×`ProductionFactor` で進捗（**線形に積む**） |
| `MegaProjectRules`（PIL-1） | 世代を跨ぐ大事業・段階建設・財政ドレイン・発起人死後の継承 |
| `ResearchRules`/`ResearchProject`（#123-127） | 研究出力・政体偏り（**抽象的な研究点**） |
| `MinistryRules`/`Ministry`（GOV-5 #158） | 省庁ツリー・`SectionalismFriction`（縦割り＝横断政策への抵抗） |
| `MilitaryFormation`/`OrderOfBattle`（#147） | 梯団ツリー（軍集団⊃軍団⊃艦隊）・司令配属 |
| `CommandStaffRules`（#885） | 提督＋副提督＋参謀の3ネームド冗長化（実効値の底上げ） |
| `Person`/`CareerPipelineRules`（LIFE-5/6/7） | 技術系の出自・席次・学閥・`TechnicalAptitude` |
| `FocusRules`（#872） | 身×口×意の同期＝3つ揃うと一時的最大出力 |
| `LogisticsRules`（GEO-3 #844） | 所有星系の連結度→国力の出し切り（版図の一体化） |

**しかし、これらは「投入を増やせば線形に成果が増える」前提**であり、人月の神話が固有に描く以下が**欠けている**：

| 人月の神話が固有に持つ視点 | 当プロジェクトでの欠落 |
|---|---|
| **ブルックスの法則＝遅れた事業に人を足すと逆に遅れる** | `ShipyardRules.Tick` は `buildPower` を足せば常に速くなる**線形**。**増員の立ち上げ遅延＋既存者の教育負担で純減する**回路が無い |
| **コミュニケーション・オーバーヘッド＝調整コストは人数のN²で増える** | `SectionalismFriction` は省益平均（人数非依存）。**チーム規模が増えるほど調整に食われる**非線形コストが無い（PERF#1117 のN²規律と思想は同じだが**生産工数には未適用**） |
| **概念の完全性（conceptual integrity）＝少数の設計者の統一思想が品質を決める** | 編制ツリーは在るが、**統括設計者の有無＝アーキテクトが居るかで成果の質が割れる**回路が無い |
| **第二システム効果＝2作目は機能過多で肥大し失敗しやすい** | `MegaProjectRules` の事業に**「過剰な野心で肥大して頓挫する」**創発が無い |
| **銀の弾丸はない＝本質的困難は単一技術で消えない／生産性の伸びは逓減** | `ResearchRules` は新技術で純粋に出力↑。**本質的複雑さは技術で消えず逓減する**という天井が無い |
| **外科手術チーム＝少数精鋭＋役割分担が大集団に勝る** | 増員＝常に正。**小さく整ったチームが大きく雑なチームに勝つ**評価軸が無い |
| **段取りの本質的困難（マイルストーンは静かに遅延する）** | 進捗は単調に積み上がる。**スリップが滑らかに/見えにくく溜まる**遅延動学が無い |

**結論**：人月の神話は当プロジェクトの**「投入＝成果」という線形前提**に、**①増員の逆効果（ブルックスの法則）②調整コストのN²③概念の完全性④逓減する生産性**という4つの**非線形の規律**を与える。これは造船#884・メガプロジェクトPIL・研究#123 を**作り直さず、共通の「工数エンジン」を一段かませる**結節点であり、**「人を足せば勝てる」を否定し高位の決断（規模/設計者/方針）の質を問う**——タイクン化回避（マイクロ操作を増やさず創発で帰結）とPERF#1117（N²を増やさない）の**ゲーム内表現**そのものになる。

## 1. 役に立つ視点（要約）

人月の神話の主張を、**本システムに効く形**で1行ずつ：

1. **人月は神話＝人と月は交換できない**。投入人数を倍にしても工期は半分にならない。→ 造船#884/メガPIL の「`buildPower` を積めば速い」線形を**逓減カーブ**へ。`LogisticsRules`（出し切れない国力）と同型の「効率係数」を生産へ。
2. **遅れた事業への増員は逆効果**。立ち上げ遅延＋既存者が新人を教える分、純生産が一時的に下がる。→ 高位の決断（増員）に**創発的な裏目**。[no-outcome-preview] 方針に一貫（事前に損益を見せない）。
3. **調整コストは人数のN²**。3人なら3経路、味方が増えるほど会議が増える。→ PERF#1117 の「ペア相互作用を増やすな」を**生産工数の式へ初めて適用**。`SectionalismFriction`#158 の規模依存版。
4. **概念の完全性が品質を決める**。多数の良い設計より、一つの統一思想。→ アーキテクト（統括ネームド）の有無で成果の**質**が割れる＝`Person`/`CommandStaffRules`#885（少数の核）に意味を与える。
5. **第二システム効果**。最初の慎ましい成功の次は、機能を盛り込みすぎて肥大し失敗する。→ メガPIL の野心レバーに**「盛りすぎ＝頓挫」**の創発。`EventEngine`#116 で発火。
6. **銀の弾丸はない**。本質的複雑さは単一技術では消えず、生産性向上は逓減する。→ `ResearchRules`#123 の生産バフに**天井（逓減）**を入れ、「技術一発逆転」を否定＝王道の地道路線を裏打ち。
7. **外科手術チーム＝少数精鋭**。役割分担した小集団が烏合の大集団に勝る。→ 「規模を抑え核を整える」が勝ち筋＝**スケーラビリティ規律（終盤ラグ回避）と同じ価値観**をゲーム目標に翻訳。

## 2. 取り入れるべきメカニクス（優先度つき・既存への接続）

> 大原則：**造船#884/メガPIL/研究#123/省庁#158 を作り直さない**。MMM はそれらに**共通の工数係数を一段かませ、接続する**だけ（additive）。
> プレイヤーの操作は「事業規模・統括設計者の有無・増員/方針」の**高位の決断のみ**、進捗・裏目はエンジン駆動・帰結は創発（タイクン化回避）。

### ★★★ 最優先（真の欠落・人月の神話の signature）

#### MMM 工数法則エンジン（人月は交換できない・逓減する有効工数）
- **`EffortRules`/`Effort`**（純ロジック・test-first）：投入「人月」`headcount × duration` を**有効工数**へ写す非線形写像。中核＝**`EffectiveEffort(headcount)` が逓減**（限界生産力が下がる）＋**調整オーバーヘッド `CoordinationOverhead(headcount)=N(N-1)/2 由来`**を差し引く。`buildPower` を1つの「工数」として束ねる。
- **接続**：`ShipyardRules.Tick`/`MegaProjectRules`（PIL-1）/`ResearchRules`#123 の進捗計算へ**有効工数係数を乗算**（基準値非破壊＝実効値パターン）。`LogisticsRules`（国力の出し切り）と同じ「係数で割り引く」流儀。**PERF#1117 のN²規律を初めて生産工数へ適用**。

#### MMM ブルックスの法則（遅れた事業への増員は逆効果）
- **増員の立ち上げ遅延＋教育負担**：進行中の事業に人を足すと、当面は**有効工数が一時的に純減**（新人は即戦力でなく、既存者が教育に取られる）。立ち上げ後に回復。遅延が大きいほど逆効果が強い。
- 決断は「増員レバー1つ」、損益は事後に判明（[no-outcome-preview]）。**「遅れているから人を足す」が罠**になる教訓を盤面化。
- **接続**：MMM-1×`EventEngine`#116（増員イベント）×`MegaProjectRules`/`ShipyardRules`。タイクン回避＝確率/損益を事前表示しない。

### ★★ 高（品質と肥大の創発）

#### MMM 概念の完全性とアーキテクト（統括設計者が品質を決める）
- 事業に**統括設計者（ネームド `Person`・高 `planning`/`engineering`）**を1人割けるか否かで、成果の**品質係数**が割れる：居れば統一思想で品質↑（崩落/手戻りリスク↓）、不在なら多人数でも整合が崩れ手戻り増。設計者は**少数の核**＝増やしても効かない（複数アーキテクトは指揮の分裂で減点）。
- **接続**：`Person`/`CareerPipelineRules`（テクノクラート）×`CommandStaffRules`#885（少数の核を整える思想）×`MegaProjectRules`の品質（PIL-2 の崩落リスクへ係数）。

#### MMM 第二システム効果（野心過多→肥大→頓挫）
- 最初の慎ましい成功の**次の事業**は機能を盛り込みすぎて肥大しやすい：野心レバーが高い＋過去に成功歴があると**仕様肥大フラグ**が立ち、工期超過・財政ドレイン増・頓挫確率↑を**創発**（`EventEngine`#116 で発火）。慎ましさ（scope抑制）が報われる。
- **接続**：`MegaProjectRules`（PIL-1）×`EventEngine`#116×`FiscalRules`（ドレイン増）。タイクン回避＝レバーは「野心/慎重」の方針1つ。

### ★ 中（生産性の天井・lore）

#### MMM 銀の弾丸はない（本質的複雑さ＝研究バフの逓減天井）
- `ResearchRules`#123 の生産性向上に**逓減天井**を導入：偶有的複雑さ（道具・自動化）は技術で消えるが、本質的複雑さ（設計の難しさ）は消えず、**生産性ブーストは積むほど効きが鈍る**。「技術一発で工数ゼロ」を否定。
- **接続**：`ResearchRules`#123×MMM-1（有効工数の天井）。**コード新設は最小**＝`ResearchOutput`→工数係数の写像に逓減を足すだけ。

#### MMM（lore）工数の神話の開示データ
- 「人を足せば速くなるという神話」「銀の弾丸を求める愚」「少数精鋭が大集団に勝つ」を**世界観 lore** として。指導者が増援を投じて却って前線が乱れる、等の歴史教訓。
- **接続**：**コード新設せず** `DisclosureLedger`（FND-4）への**lore データ入力**。CCX-6（世界観codex退避）方針に一貫。

### ❌ 不採用（重複・タイクン化・後退）

| 不採用 | 理由 |
|---|---|
| メガプロジェクト基盤そのもの（段階建設・財政ドレイン・継承） | **PIL-1（#1090）がカバー**。MMM は工数係数を**かませるだけ**・作り直さない |
| 工期と品質のトレードオフ（急造→崩落） | **PIL-2（#1091）がカバー**。MMM-3 はアーキテクト品質係数を**接続**するのみ |
| 技術者が工法を保持し移籍で伝播 | **PIL-3（#1092）がカバー**。MMM-3 は「統括設計者の有無」のみ・伝播は触らない |
| 省庁の縦割り摩擦そのもの | **`SectionalismFriction`#158 がカバー**。MMM-1 は**規模依存のN²**を生産工数へ足すだけ |
| 個別タスク/工程表/ガントの微操作 | **タイクン化回避の大原則に反する**。決断は高位レバー（規模/設計者/方針）のみ・進捗はエンジン駆動 |
| ソフトウェア固有の用語・実装（バージョン管理/デバッグ/言語論） | 世界観が違う。**工数法則の構造のみ**輸入し、星間国家の生産（建艦・要塞・研究）へ写す |

### 推奨着手順
`MMM-1`（工数法則エンジン＝最も固有で欠落の大きい signature・以後の土台）→ `MMM-2`（ブルックスの法則＝増員の逆効果・MMM-1 直依存）→ `MMM-3`（概念の完全性＝品質の核・既存 Person/#885 活用）→ `MMM-4`（第二システム効果＝メガPILの肥大創発）→ `MMM-5`（銀の弾丸＝研究天井・最小）→ `MMM-6`（lore）。
