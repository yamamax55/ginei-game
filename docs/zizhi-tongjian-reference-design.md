# 資治通鑑 参考設計（EPIC #ZTGJ）

> 参照元：司馬光『資治通鑑』（北宋・編年体の通史）。書名は「治を資（たす）くる、通（つう）じての鑑（かがみ）」＝**統治を助けるための、通時的な鑑（鏡）**。歴代の君臣の成敗（成功と失敗）を**年代順に一本の編年史**へ束ね、要所に編者の論評（「臣光曰く」）を挿んで**何が国を興し何が滅ぼしたかの教訓**を示す——為政者が**過去の類似事例を鏡として現在の決断を正す**ための、君臣の成敗事例集（イベント素材の宝庫）。
> 本ドキュメントは、当プロジェクト（Ginei＝銀英伝風の星間国家戦略＋既に巨大な政治・群像・イベント・開示の純ロジック層）にとって**役に立つ視点**だけを抽出し、EPIC `#ZTGJ` として issue 化する提案。
> 著作権注意：固有名・文章・人物・固有設定は流用せず、**「鑑（先例で決断を正す）」「君臣の成敗の教訓化」「納諫の度量」「才徳の人物鑑識」「編年史＝事例の問い合わせ可能な台帳」という構造パターンのみ**を参考にする。架空の勢力・人物で実装する。

## 0. なぜ「資治通鑑」が本システムに役立つか

当プロジェクトは「決断・人物・イベント・記録・正統性」に関わる純ロジックを既に大量に保有している（[CLAUDE.md] 参照）：

| 既存（決断・人物・記録・歴史） | カバー範囲 |
|---|---|
| `DecisionQueue`/`PendingDecision`/`DecisionTriageRules`（決裁デスク） | 非ブロッキングの決裁キュー（諮問/建白結果/イベントの裁可・放置=AI自動解決） |
| `EventEngine`/`GameEventDef`/`EventRules`（#116） | データ駆動の条件発火→通知/選択肢→効果 |
| `PersonRules`/`Person`（正名 #866） | 軍人/文民の役割×役職の適材適所（軍才/文才の適性） |
| `DisclosureRules`/`DisclosureLedger`（FND-4 #495） | 確定的な連鎖開示（秘史A→秘史B、不動点まで） |
| HRDT-1/2 証言・正史（#1702/3） | 出典つき証言・異説併記・勝者が書く正史（歴史記述の認識論） |
| MEYASU 目安箱（越階の諫言回路） | プレイヤー＝建白↑/諮問裁可↓/注入横の越階チャネル |
| LEAR 道化の特権（#1175） | 制度外の者だけが罰されず真実を言える（諫言の迂回路） |
| `DynastyRules`/`Regime`（#867）・`JusticeRules`（#918） | 王朝サイクル・天命・正統性／複数の正義観で行動の是認が変わる |
| NOTIF `NotificationCenter` | 通知の単一窓口・有界リングバッファ |

**しかし、資治通鑑が固有に描く「鑑（先例の鏡）としての統治＝過去の成敗を教訓として現在の決断に作用させる学習装置」**は無い：

| 資治通鑑が固有に持つ視点 | 当プロジェクトでの欠落 |
|---|---|
| **鑑＝過去の類似事例を鏡として現在の決断を正す** | `DecisionQueue` は決断を提示するだけ。**現決断に類似の過去事例（先例）を引き当てて添える**回路が無い。HRDT は記録の認識論だが、**記録を"決断の助言"として引き当てる**ことはしない |
| **君臣の成敗を教訓化＝結果に「なぜ興/滅したか」のラベルを付けて台帳化** | 事件は起きて流れるだけ。**「この決断はこう転んだ→成敗の教訓」として確定し、後の先例プールへ蓄積する**学習層が無い |
| **納諫の度量＝君主が諫言を容れるか拒むかが結果を分ける**（兼聴 vs 偏信） | MEYASU/LEAR は**諫言の届く経路**を作る。だが**届いた後、君主（勢力AI/元首）がそれを容れる度量を持つか**＝納諫度が成敗を左右する回路が無い |
| **才徳の人物鑑識＝才（能力）と徳（品性）を分けて人を量る**（司馬光の有名な四類型：聖人/君子/愚人/小人） | `PersonRules` は軍才/文才の**適性**。だが**才と徳を別軸で持ち、才あって徳なき者の登用が後で害をなす**（用人のジレンマ）が無い |
| **編年史＝全事件を一本の時系列へ束ね問い合わせ可能にする** | 統一時間 `GameClock` はあるが、**起きた成敗を年代順の台帳へ束ね「似た状況」を検索する**事例DBが無い。HRDT-1 は証言の信頼度、こちらは**決断助言のための事例索引** |
| **借鑑＝「前車の覆るは後車の戒め」＝失敗の反復を制度的に避ける** | 同じ轍を踏んでも警告が出ない。**過去の失敗パターンに照らして現決断に警告/推奨を出す**諮問が無い |

**結論**：資治通鑑は当プロジェクトに**「統治とは過去の成敗を鏡として学ぶ営みである」という鑑（ミラー）の思想**を与える。`DecisionQueue`（決裁デスク）の隣に**「先例を引き当てて決断に添える助言層」**を置き、起きた決断の結果を**成敗の教訓として台帳化**して次の先例プールへ回す——**決断→結果→教訓→次の決断の助言**という学習ループを敷く。さらに**納諫の度量**（容れる君主は栄え拒む君主は滅ぶ）と**才徳の人物鑑識**（才あって徳なき登用の罠）を加える。HRDT（記録の認識論）が「真実は視点で割れる」を、ZTGJ は「**割れる記録から教訓を抽出し決断に効かせる**」を担い、両者は**記録↔活用**で住み分ける。決裁デスク・イベント・人物・開示が**一段"賢く"なる**のが最大の収穫。

## 1. 役に立つ視点（要約）

資治通鑑の世界観を、**本システムに効く形**で1行ずつ：

1. **統治は過去の成敗を鏡として学ぶ営み**（鑑）。同じ決断は歴史に先例があり、それを引けば賢く決められる。→ `DecisionQueue`（決裁デスク）に**先例の引き当て**を足す＝決断が孤立せず歴史と接続する。
2. **君臣の成敗には理由があり、教訓化できる**。決断の結果は「なぜ興/滅したか」のラベルで台帳化され、次の先例になる。→ `EventEngine`#116・`DisclosureLedger`#495 の隣に**成敗の事例台帳**＝決断→結果→教訓の学習ループ。
3. **明君は諫を容れ、暗君は諫を拒む**（納諫の度量＝兼聴 vs 偏信）。届いた諫言を容れるか否かが成敗を分ける。→ MEYASU（諫言経路）/LEAR（道化）の**先**＝届いた後の**受容度**を勢力の統治特性に。
4. **才と徳は別物で、才あって徳なき者は害をなす**（司馬光の用人論）。能力で登用すると品性で裏切られる。→ `PersonRules`#866 に**徳の軸**＝登用の創発的リスク（有能な佞臣・離反・腐敗）。
5. **編年史は全事件を時系列に束ね、似た状況を検索できる**。→ 統一時間 `GameClock` 上の**事例索引**＝「前車の覆るは後車の戒め」を機械化（借鑑）。
6. **「臣光曰く」＝編者の論評が教訓を名指す**。出来事に編者の価値判断（正義観）が添う。→ `JusticeRules`#918 で**事例の教訓は正義観により評価が割れる**＝唯一の正解を押し付けない。

## 2. 取り入れるべきメカニクス（優先度つき・既存への接続）

> 大原則：**`DecisionQueue`/`DecisionTriageRules`・`EventEngine`#116・`PersonRules`#866・`DisclosureLedger`#495・HRDT-1/2・MEYASU・LEAR を作り直さない**。ZTGJ はそれらに**「鑑（先例で決断を正す学習ループ）」「納諫の度量」「才徳の人物鑑識」**を足し、接続するだけ（additive）。タイクン化回避＝先例引き当て・教訓化・納諫はいずれも**高位の決断とその創発的帰結**で、プレイヤーに台帳のミクロ編集をさせない（自動で添い、自動で蓄積し、結果は事後に判明＝`no-outcome-preview`）。

### ★★★ 最優先（真の欠落・ZTGJ の signature）

#### ZTGJ 先例の引き当て（鑑＝過去の類似事例を現決断に添える）
- `PrecedentRules`（static・test-first）＝事例台帳（ZTGJ-2）から**現在の `PendingDecision` に状況が類似する過去事例**を引き当てる純関数。類似度＝決断カテゴリ／関与勢力の状態（安定度・正統性・財政）／規模の近さ。引き当てた先例（「過去、似た状況でAを選んだ勢力はこう転んだ」）を決断に**助言として添える**＝鏡。**結果を予言しない**（過去の事例の傾向を示すのみ・現決断の帰結は不確実）。
- 接続：`DecisionQueue`/`PendingDecision`（`DecisionSource.諮問` の助言として `body`/`imageKey` に添う）・ZTGJ-2 事例台帳・HRDT-1（証言の信頼度＝先例の確かさにも効く）。**`no-outcome-preview`（決断前に勝敗を見せない）と一貫**＝予想でなく「過去の傾向」。

#### ZTGJ 成敗の事例台帳（君臣の成敗を教訓化し次の先例へ＝学習ループ）
- `CaseLedger`（plain・headless・有界）＋`CaseRules`（static・test-first）＝決断が結果を出すと**事例エントリ**（状況スナップショット＋採った選択＋転帰＝興/滅/中立＋教訓ラベル）を台帳へ記録。台帳は**先例プール**として ZTGJ-1 が検索する。「前車の覆るは後車の戒め」＝失敗事例が後の同種決断に**警告**として効く（借鑑）。**有界リングバッファ＋カテゴリ集約で無制限増加を防ぐ**（PERF規律・打ち切りは log で明示）。
- 接続：`DecisionQueue`（決断の転帰を記録）・`EventEngine`#116（イベント結果も事例化）・`DisclosureLedger`#495（歴史 lore の隣に成敗の実例）・`NotificationCenter`（教訓＝出典つき通知）。**決断→結果→教訓→次の助言の学習ループの核**。

### ★★ 高（納諫の度量・才徳の人物鑑識）

#### ZTGJ 納諫の度量（諫を容れる君主は栄え、拒む君主は滅ぶ＝兼聴 vs 偏信）
- `RemonstranceRules`（static・test-first）＝MEYASU の建白／LEAR の道化／参謀の諫言が**届いた後**、君主（勢力AI/元首 `Person`/`ICharacter`）が**容れるか拒むか**を決める統治特性「納諫度」を導入。納諫度が高い＝兼聴（広く容れて誤りを正す→正統性/安定度に好影響）、低い＝偏信（佞臣だけ信じ良諫を退ける→腐敗・離反・後の失敗事例化）。**諫言を容れた/拒んだ結果は事例台帳（ZTGJ-2）へ**＝納諫の成否も教訓になる。
- 接続：MEYASU（建白の受容判定）・LEAR-2（佞臣の逆選択の解毒＝納諫が高ければ道化の真実が効く）・`DynastyRules`#867（拒諫＝正統性の蝕み）・`PersonRules`#866（元首の徳の一面）。**プレイヤー越階の建白が"容れられるか"に納諫度が効く**＝MEYASU を一段リアルに。

#### ZTGJ 才徳の人物鑑識（才と徳を分けて量る＝用人のジレンマ）
- `PersonRules`#866 に**徳の軸**（品性・忠誠の素地）を追加し、`TalentVirtueRules`（static・test-first）＝才（能力）と徳（品性）の二軸で人物を**四象限に鑑識**（才徳兼備＝聖賢／徳勝才＝君子／才徳薄＝愚人／才勝徳＝小人＝**有能な佞臣**）。才で登用すると徳の欠落が**後で害をなす**（離反・腐敗・拒諫の助長・クーデター #215 の素地）＝登用の創発的リスク。基準値非破壊（実効値パターン）。
- 接続：`PersonRules`#866（適性に徳軸を追加・read-only拡張）・`GovernmentRegistry`/`OfficeRules`（任命時の鑑識）・`CivilianControlRules`#145（小人の登用＝クーデターリスク）・ZTGJ-3（小人を容れる君主＝納諫の偏信）。

### ★ 中（編年史索引・教訓の正義観・lore）

#### ZTGJ 編年史索引（全事件を時系列へ束ね「似た状況」を検索＝借鑑の機械化）
- `ChronicleIndex`（plain・headless）＝統一時間 `GameClock` 上の事例台帳（ZTGJ-2）を**年代順＋カテゴリで索引化**し、ZTGJ-1 の先例引き当てを**高速・決定的**にする薄い索引層（派生値は変化時のみ再計算しキャッシュ＝PERF規律）。事例DBそのものは ZTGJ-2、索引は引き当ての効率化。**新規 state を増やさず ZTGJ-2 の上に索引を張るだけ**。
- 接続：`GameClock`（統一時間）・ZTGJ-2 事例台帳・ZTGJ-1 引き当て。**シミュLOD**＝遠方/古い事例は粗く集約（終盤ラグ回避）。

#### ZTGJ 教訓の正義観（「臣光曰く」＝編者の論評は正義観で割れる）
- 事例の教訓ラベル（成/敗/可/否）は**唯一の正解でなく**、`JusticeRules`#918 の各正義観（功利/ロールズ/リバタリアン等）で**評価が割れる**。同じ決断が「功では成・徳では敗」と分かれる＝編者の価値判断が添う「臣光曰く」の構造。プレイヤーに**唯一の教訓を押し付けない**（HRDT「勝者が正史を書く」とも響く）。コード新設は最小（ZTGJ-2 の教訓を `JusticeRules` で再評価するだけ）。
- 接続：`JusticeRules`#918・ZTGJ-2 事例台帳・HRDT-2（正史の確定）。

#### ZTGJ（lore）世界観の開示データ（鑑・前車の戒め・才徳）
- 「治を資くる鑑＝統治は歴史に学ぶ」「前車の覆るは後車の戒め」「才ある小人は害をなす」「兼聴すれば明、偏信すれば暗」＝開示データ。**コード新設せず** `DisclosureLedger`（FND-4）への lore データ入力。CCX-6（世界観 codex 退避）方針に一貫。
- 接続：`DisclosureLedger`（FND-4）・ZTGJ-2（成敗の実例が lore を裏打ち）。
