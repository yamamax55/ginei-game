# 並列エージェント実装ワークフロー（Core-first 契約 → disjoint 並列 → 親が配線）

> 大きめの実装を**速く・安全に・安く**進めるための標準手順。  
> 教訓：密結合（同じホットファイルを多数が触る）作業をフル並列にすると、**ベースずれ・マージ衝突・統合時まで破綻が見えない**問題が起きる。  
> 原則 ＝ **並列は「独立な広がり」に。配線（共有ホットファイル）は親が逐次に。**

## 0. 手法の選択（まずここを判断）
| タスクの性質 | 手法 |
|---|---|
| 独立・ファイル分離（新Coreモジュール／別サブシステム／UI overlay／演出／ドキュメント） | ✅ disjoint 並列（2〜3体） |
| 密結合・同じホットファイルを多数が編集（例：会戦の与ダメ修飾子） | ❌ 並列にしない。**親が逐次**（1ブランチ）で実装 |

判断基準：**触るファイル集合が重なるなら並列にしない。** 重なる部分は必ず親が持つ。

## 1. 親：Core 契約を凍結（最初の1コミット）
- 並列前に、親が**Core の API/シグネチャ（インターフェイス）だけ**を先に確定してコミットする。
  - 例：`public static float XxxRules.Factor(...)` の引数・戻り値・enum 定義。中身は最小スタブでよい。
- これで「共有される契約」が固定され、各エージェントは**安定した型**に対して実装できる＝衝突源が消える。
- 同時に**ファイル所有権の割り当て表**を作る（誰がどのファイルを作る/触るか）。**所有が重複しないこと。**

## 2. 親：disjoint な並列エージェントを起動（2〜3体まで）
各エージェントへの**必須指示**：
- **ベース SHA を明示**：「`<SHA>` から `git checkout -b claude/<task>` せよ」。曖昧にしない（古いベース事故の予防）。
- **所有ファイルのみ編集**：割り当てられた**新規ファイル中心**。共有ホットファイル（FleetWeapon/FleetAI/FleetStrength/BattleManager/ShipCombat 等）は**触らない**＝配線は親がやる。
- 規約：Core は test-first（EditMode）。新規 `.cs` に必ず `.meta`。実効値パターン。日本語コメント。スケーラビリティ（部隊単位・間引き・N²回避・ホットパス無確保）。
- `CLAUDE.md` は編集しない（親が統合）。push/PR しない。
- コミット末尾に必ずセッションURL。モデル識別子を入れない。
- 完了報告：**ブランチ名・`git rev-parse HEAD`・追加/変更ファイル一覧・概要**。
- 台数は **2〜3体**・**ファイル非重複**を厳守（7体×ホットファイルは過剰）。

## 3. 親：マージ前ゲート（必ず確認）
- **ベース検証**：`git merge-base <agentSHA> <feature>` が想定ベースか。ずれていたら git-merge しない（既存編集が現行を退行させる恐れ）。
  - ずれた場合：**新規ファイルは `git checkout <SHA> -- <newfiles>`、既存編集は手動再適用**。
- **重複定義チェック**：同名 class/enum が複数ファイルに無いか。
- **外部API実在チェック**：エージェントが呼んだ既存メソッドのシグネチャを `grep` で照合。

## 4. 親：統合（配線）と検証
- 正ベースかつ非重複なら `git cherry-pick`／`git merge`。
- 共有ホットファイルへの**配線は親が逐次**で行う（各機能の倍率を `ShipCombat.ComputeDamage` 等へ）。
- `CLAUDE.md`（test-first 一覧・コンポーネント表）を**親が一括更新**。
- ローカルに dotnet/Unity が無ければ**目視＋grep で型整合を確認**し、テスト実行は CI 任せ（失敗 webhook で対応）。
- worktree/一時ブランチは統合後に `git worktree remove --force` ＋ `git branch -D` で掃除。
