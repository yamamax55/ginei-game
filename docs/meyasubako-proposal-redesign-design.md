# 目安箱＝越階の諫言回路 — 提案システム再設計（EPIC #MEYASU）

> 戦略レイヤーの意思決定操作（提案システム）を、**「プレイヤー＝目安箱という越階の諫言回路」**へ再設計する。
> 既存の社内ワークフロー設計 [`corporate-workflow-strategy-ui-design.md`](./corporate-workflow-strategy-ui-design.md)（WF-1〜5）の **大型アップグレード**。WF の状態機械（`WorkflowRules`/`Proposal`）を**作り直さず、その上に**「プレイヤーの立ち位置の確定」「LLMによる稟議生成・決裁」「個人ごとの信認」「建白の伝播」を足す。
> 状態：設計メモ（実装未着手）。数値・創作裁定は【要・作者判断】。

## 0. なぜ再設計するか — WF §7 が残した宿題の解決

既存 WF 設計は稟議の**状態機械**（起案→合議→決裁→執行→通知）を `OfficeRules`/`MinistryRules`/`CivilianControlRules` へ接続する優れた骨格を持つ。だが §7【要・作者判断】に**未決の核**が残っていた：

> 「**プレイヤーの立ち位置**：元首として全決裁するのか、一省庁の長として権限内だけ動かすのか」

この二択はどちらも問題を抱える：

| 既存の二択 | 問題 |
|---|---|
| **元首として全決裁**（頂点） | god mode。微操作＝タイクン化（本作の最大の禁忌）。「君は物語の主人公ではない」と矛盾 |
| **一省庁の長**（序列内の一ノード） | 権限の壁が固定的で、国家全体への関与感が薄い。立ち位置が一方向に固定 |

本作の**逆ルート要求**（プレイヤーが起案して上位の決裁を仰ぐ）と**順ルート要求**（上位がプレイヤーへ裁可を仰ぐ）を**同時に**満たそうとすると、序列内のどの座席に置いても矛盾する（上でもあり下でもある、になる）。

### 解決：プレイヤーを序列の「外」に出す＝目安箱という制度

プレイヤーを役職ではなく**「越階の諫言回路（目安箱という制度）」**にする。箱は誰の上でも下でもなく、**全員がアクセスできる側回路**。これで上下の矛盾が消え、両ルートが共存する：

```
        ┌──────────────── 目安箱（プレイヤー＝制度・序列の外）────────────────┐
        │                                                                       │
   ① 建白（下→箱→上）          ② 諮問・裁可（上→箱）          ③ 注入（箱→任意）
   末端が正規ルートを越階して     最高権力者が決めきれない        プレイヤーが異端の
   投書。プレイヤーが汲む/黙殺。   重大案件を箱へ問い返す。        アイデアを起草して放つ。
   汲めば上へ流す。              プレイヤーが裁可。拒否に代償。
        │                                                                       │
        └───────────────────────────────────────────────────────────────────────┘
                                       ↑↓
                  官僚機構（ネームドたち＝省益/閥/正統性を持つLLMエージェント）
                  └ OrderOfBattle（軍）/ Ministry（文）のツリーをそのまま流路に
```

- **箱は「役職」ではなく「制度」**＝本作が得意とする `Organization`（制度化／カリスマの日常化）と同じ発想。突き詰めれば**天命／世論のオラクル**（`DynastyRules` の天命を、官僚が重大案件で「箱に伺いを立てる」形にできる）。
- これは銀英伝の官僚機構（帝国の省庁／同盟の議会）を操作系にする WF の狙いを**さらに先鋭化**する。

## 1. 設計の核（4本柱）

### 柱A：プレイヤー＝越階回路（序列外の制度）
- プレイヤーは序列内の座席を持たない。**箱という側回路**として、建白を汲み（↑へ流す）／諮問に裁可し／注入する。
- 既存 `WorkflowRules` の状態機械に**「箱」を新しいアクター種別**として接続するだけ（並行システムを作らない）。
- 政体で箱の効き方が変わる（後述・柱と直交）。

### 柱B：LLMはWHY、CoreはWHAT（決定論の分離）
- **LLMが生むのは "WHY"**＝稟議の起案文・決裁の判断・口調・説得。ネームドの人格が文面に出る。
- **Coreが握るのは "WHAT"**＝実行可否の検証・効果適用・状態遷移。LLMに状態を直接書き換えさせない。
- **決定論とセーブの分離**：稟議の**文面はエフェメラル（再生成可・保存しない）**、その下の**構造化された決定だけを `effectKey` 駆動で決定論的に保存**（WF の効果レジストリを踏襲）。テスト（EditMode/TestHarness）はCore側だけ見ればよい。
- 接続：ローカルLLM #909・ルールAIフォールバック #915（LLM不在でもテンプレで回る）・`CampaignSerializer`（状態↔JSON）。

### 柱C：個人ごとの信認（借り物の権威＝唯一のリミッター）
- 箱が聞いてもらえるのは**「借り物の権威」**（`ConsentRules`/`Polity`＝権力は借り物）。これを**唯一のゲージ**にする＝人工的なマナbarを置かない。
- **信認はネームド個人ごと**に持つ。「この男には顔が利くが、あの参謀には警戒されている」が出る＝人事・閥・列伝と直結。

| 出来事 | 信認 |
|---|---|
| 黙殺された建白が後に正しかったと判明 | ↑（以後その相手に通りやすい） |
| 諮問の裁可を拒否（＝代償） | ↓ |
| 乱発・的外れ・盛った建白がバレる | ↓ |
| ある相手の信認が枯れる | その相手は箱を**壁紙化**（読まなくなる）／全体で枯れると国家規模のオラクル失墜 |

- **スケーラビリティ規律（PERF #1117）**：全ネームド分を毎ティック回さない。**箱が一度でも触れた相手だけ疎に持つ（lazy/sparse、`NotificationCenter` の有界リング思想）**。
- 接続：`ConsentRules`/`Polity`・正統性 `Regime`・人物/列伝 #784・閥 `CareerPipelineRules.CliqueBond`。

### 柱D：建白の伝播（多エージェント官僚機構を提案が通過する）
- 投書は自動実行されない。鎖の中で各ネームドの**省益・閥・正統性**の検査を通らないと上に行かない。**大半は勝手に死ぬ**＝これが内生的スロットル（特別な通貨が不要）。
- 提案は通過の過程で**汲まれる／握り潰される／歪んで伝播する／黙殺後に再浮上する**。
- **生存・伝播のモデルは Core（安いロール）**、**文面の歪み・口調は LLM**。両層を分離する。
- 接続：`MinistryRules.SectionalismFriction`（省益＝抵抗）・`CliqueBond`（閥が担ぐ/潰す）・正統性。

## 2. プレイヤーの3つの動詞（傍観者化しない設計）

純粋な目安箱（投げて見てるだけ）は傍観者ゲーム化する。能動性は次の3動詞で担保する：

| 動詞 | 内容 | プレイの腕 |
|---|---|---|
| **建白（注入↑）** | 異端のアイデアを起草し、誰の箱に・いつ投げるか | **ターゲティング**（腐敗した上司なら握り潰される／改革派なら担がれる）＋**フレーミング**（盛れば疑われる）＋**タイミング**（危機で通りやすい） |
| **裁可（諮問↓）** | 上が詰んで箱に投げ返した重大案件を承認/却下 | **どの一件に信認を賭けるか**の選択。拒否は代償だが、信念を貫く手段 |
| **人事（間接）** | 直接任命はできないが「あの男を登用せよ」と建白を重ねて通せば回路自体が変わる | 同調者を据える＝以後上がる建白の数・論調が変わる（戦う稟議が減る） |

→ **命令はできない、でも種は撒ける。** 撒いた種が無名の手を経て政策になる——これがチ。（#1254・真理のリレー）のゲーム化であり、本作の核「君は物語の主人公ではない」の操作面での体現。

## 3. 政体による差別化（WF §3 を継承・箱の効き方が変わる）

同じ箱でも政体で挙動が変わる＝抽象的な政治シミュ数値（`政体`/`inclusiveness`/`Regime`）を**操作の自由度として毎ターン体感**させる。

| 政体 | 建白（↑） | 裁可（諮問↓） | 代償 |
|---|---|---|---|
| **専制（帝国型）** | 親裁が通りやすい・段数少 | 裁可は形式的 | 一回ごとに正統性が急減（易姓革命リスク・`DynastyRules`） |
| **立憲** | 副署が要る＝閣僚が拒否可 | 裁可拒否は憲政危機 | 安定だが不自由（`ConstitutionRules`/`MagnaCartaRules`） |
| **民主（同盟型）** | 議会・政党の同意が要る（`PartyRules`） | 諮問は稀（合議が先に決める） | 遅いが堅い |

→ WF §3 の「速いが脆い専制 vs 遅いが正統な民主」を、**箱の通りやすさ**として表現。

## 4. データモデル案（Core・test-first・WF を拡張・並行新設しない）

> WF の `Proposal`/`ProposalStatus`/`WorkflowRules`/効果レジストリを**土台に流用**。本再設計が足すのは「箱アクター」「信認」「伝播の生存ロール」「諮問→箱 defer」の4点。LLM層は別ファイル（Game/Data）で、Coreは read-only。

### 4-1. `BoxCredibility`（Core・`[Serializable]`・sparse）
```
faction         : 勢力
entries         : Dictionary<characterId, float>  ← 個人ごと信認 0..1（触れた相手だけ・lazy）
globalDeference : float（国家規模の箱への傾聴度＝オラクル失墜の指標）
```
- `CredibilityRules`（static・唯一の窓口）：`Of(id)`（無記録＝既定値）／`Adjust(id, delta, reason)`／`Decay`（不使用で中庸へ収束）／`Heard(id)`（信認で建白の通過率に係数）。基準値非破壊・実効値パターン。

### 4-2. `Petition`（Core・WF `Proposal` の派生 or フィールド追加）
- WF `Proposal` に **`origin`（建白/諮問/注入）**・**`carrierId`（今この稟議を担いでいる中継者）**・**`hops`（伝播履歴＝誰の手を経たか）**を足す（列伝・チ。リレーと接続）。
- `effectKey`＋効果レジストリは WF のまま（決定論セーブ）。

### 4-3. `PetitionFlowRules`（static・Core・伝播の生存モデル）
- `SurvivalChance(petition, carrier, ministries)` … `SectionalismFriction`＋`CliqueBond`＋正統性から「次の階へ通るか」の確率（決定論 roll）。
- `Advance(petition, roll)` … 通過/握り潰し/歪み（歪みの内容はLLM層が後付け）/黙殺。
- `Resurface(petition)` … 黙殺された建白が後に再浮上する条件（事後の正しさ判明＝信認↑のトリガ）。
- 政体差は `inclusiveness`/`Regime` を引数で受けて分岐（`FactionState` read-only）。

### 4-4. `WorkflowRules` 拡張（諮問→箱 defer）
- `ShouldDeferToBox(proposal, government)` … 重大案件かつ閣内不一致/正統性低下で**箱へ諮問**する終端を追加。
- `ApplyRatification(proposal, approved)` … 裁可結果を反映＋拒否時に `CredibilityRules`/正統性へ代償。

### 4-5. LLM層（Game/Data・Coreは不参照・#909/#915）
- `RingiDrafter`（起案文生成）／`RingiAdjudicator`（決裁の判断＝LLMだが**最終可否はCore検証が上書き**）。出力は文面のみ、決定は `effectKey`/構造化フィールドへ落とす。
- **LLM-LOD**：建白は**まず安い `PetitionFlowRules` の生存ロール**で大半を間引き、**浮上した数件だけフル生成**（`AutoBattleSim` の抽象解決と同思想）。LLM不在時はテンプレ文（#915 フォールバック）。

## 5. UI（UITK・WF-2 のワークトレイを発展）

WF の「ワークトレイ（受信箱）／起案フォーム／決裁画面／ダッシュボード」を流用しつつ、**箱の受信箱**として再構成：

| 画面 | 中身 | 既存の手本 |
|---|---|---|
| **箱の受信箱** | 建白（↑汲む/黙殺）・諮問（↓裁可/却下）の一覧。**各行に相手の信認を可視化**（顔が利く/警戒されている） | `OrderOfBattlePanel`／`NotificationFeed` |
| **建白フォーム** | 誰の箱に・何を・どう書くか。**ターゲット相手の信認と通りやすさを示す**（が結果は不確実＝[[no-outcome-preview]]） | WF 起案フォーム |
| **裁可画面** | 諮問の詳細＋伝播履歴（誰の手を経たか）＋［裁可］［却下］。拒否の代償を**事前に数値で見せない** | `StrategyEventPanel.Show` |
| **伝播ビュー（任意）** | 撒いた建白が官僚機構をどう流れ・歪み・再浮上したかの足跡（チ。のリレー可視化） | — |

- 表示中 `Time.timeScale=0`・`RuntimeInitializeOnLoadMethod` で戦略シーン自動生成・キーは `GameInput` に追加（既存パネル流儀）。
- **[[no-outcome-preview]] の規律**：建白の起案文は「説得」であって「予測表」ではない。盛った稟議に騙されることもある。承認/裁可の余波は**事後に `NotificationCenter`** へ。

## 7. 完了条件（EPIC）
- プレイヤーが**序列外の目安箱**として、建白（↑）・裁可（↓・諮問）・注入（横）の3動詞で関与でき、上下の立ち位置矛盾が無い。
- **個人ごとの信認**が唯一のリミッターとして機能し、「なんでもできないが、箱に放り込めばある程度は動かせる」が成立（god mode でも傍観者でもない）。
- 建白が**多エージェント官僚機構を伝播**し、握り潰し/歪み/黙殺/再浮上が創発する。
- **LLMはWHY・CoreはWHAT**が分離され、文面はエフェメラル・決定は決定論的に保存・テストはCoreで担保。終盤ラグを生まない（LOD）。
- いずれも WF・既存政治シミュ（`OfficeRules`/`MinistryRules`/`ConsentRules`/`DynastyRules` 等）を**後退させず接続**する additive 設計。

## 8. 【要・作者判断】
- **信認の具体カーブ**：黙殺→事後正解での回復量、裁可拒否の減少量、枯渇＝壁紙化の閾値。
- **諮問トリガの厳しさ**：上が箱へ defer する条件（閣内不一致/正統性低下のどこから）。裁可を希少で重くする度合い。
- **伝播の段数**：建白が何階を通るか（政体別）。再浮上の頻度（黙殺の悔しさをどこまで残すか）。
- **LLM運用**：#909 ローカルLLM の導入時期。それまではテンプレ（#915）で骨格を回すか。
- **箱の格（確定済み＝越階回路）**：純・建白者ではなく**諮問＋注入を持つ越階回路**で行く（本メモの前提）。
