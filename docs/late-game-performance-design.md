# 終盤スケーラビリティ設計（EPIC #PERF — 「ステラリスにならない」）

> グランドストラテジーの宿痾＝**終盤ラグ**を、設計規律として明文化し守る。参照する反面教師は Stellaris（Paradox）の late-game 性能劣化。
> 本ドキュメントは性能の**アーキテクチャ EPIC**（area:infra）。参考作品メカニクス（§5-2）とは別系統。

---

## 0. 診断：なぜグランドストラテジーは終盤で重くなるか

Stellaris 終盤ラグの本当の原因は、ひとことで **「個体（pop）を、際限なく増える数だけ、毎ティック全部再計算するから」**。分解すると4因子の掛け算：

| 因子 | Stellaris での現れ | 一般化した罠 |
|---|---|---|
| **個体粒度の計算** | 経済を pop 単位で計算（職割り当て・幸福度・移民…） | コストが O(個体数) |
| **際限ない増加** | pop は帝国×惑星×成長で超線形に増え、GCされない | 母数が単調増加 |
| **毎ティック全再計算** | 差分でなく状態をゼロから導出し直す | 変わってない所まで再計算 |
| **N² 相互作用** | 外交opinion・交易路・諜報が帝国²／毎ティックpathfind | ペア数の爆発 |
| **直列ティック** | 依存ある経済計算が単スレ＝最遅系がフレームを律速 | 並列化できない |

**結論**：コストが個体数に比例し、個体数が際限なく増え、毎ティック全再計算、しかも直列——4つが掛け算で効いて終盤に崩れる。

## 1. 本システムの立ち位置：規律が既に効いている

**重要：当プロジェクトの「タイクン化回避」は、そのまま「ラグ回避アーキテクチャ」である。同じ決断の裏表。**

| Stellaris の罠 | 当プロジェクトの既存規律（＝反ラグ） |
|---|---|
| pop 単位経済 | `FactionState`＝**勢力ごと（数個）**／内政は `Province`/惑星**集約**（#767 ハイブリッド・タイクン回避） |
| 毎ティック全再計算 | `GovernanceRules` は**目標値へ収束**（安い差分）／`Squadron` 陣形スロット・`BeamFx` グラデは**変化時のみ再計算しキャッシュ** |
| 毎フレーム駆動 | `CalendarDispatcher` の日/月/年**境界**で経済Tick／`TimeFlowRules` 圧縮 |
| 見てない所も全解像度 | `AutoBattleSim`＝**観てない会戦はランチェスター積分で抽象解決**（シミュLODの種） |
| 無制限リスト | `NotificationCenter` は**リングバッファ（Capacity）**／`DamagePopup.MaxActive`=24 上限 |
| `FindObjectsByType` 乱用 | `FleetRegistry` 単一在庫に集約 |

→ つまり当プロジェクトは**ステラリスの三重苦を構造的に持ち得ない設計**を、すでに選んでいる。本EPICは新機能でなく、**その規律を明文化し、今後の追加（特に新Tick系・新カップリング）で守らせる**ためのもの。

## 2. 守るべき原則（5箇条）

1. **個体粒度へ降りない**：pop/個艦経済を作らない。集約（勢力/星系/惑星）に留める。realism 欲しさの粒度低下が終盤ラグの第一原因。
2. **毎フレームでなく暦境界でTick**：マクロ進行は `CalendarDispatcher` に相乗り。`Update` で毎フレーム全再計算しない。
3. **差分・収束・キャッシュ**：状態は小さなdeltaで進め、派生/集約値は**変化時のみ**再計算してキャッシュ。
4. **N² の相手数を増やさない**：ペア相互作用（外交/関係/カップリング/裁定）は**遅延・オンデマンド・間引き**で、毎ティック全ペア走査しない。勢力数を少なく保つ。
5. **シミュLOD＝見ていない物は粗く**：遠方/非係争/オフスクリーンは粗い粒度でTick・さらに集約。観測解像度とシミュ解像度を分離する。

## 3. EPIC #PERF の子Issue（着手順）

> 純ロジック（LOD/ティック方針/キャッシュ無効化判定）は test-first（EditMode/TestHarness）。計測・監査系は実機プロファイルで担保。

> **EPIC = #1117**。GitHub issue 起票済み（#1118〜#1124）。

| # | issue | タイトル | 主眼 |
|---|---|---|---|
| **PERF-1** | #1118 | 性能予算とプロファイル基盤（Tick/フレームのコスト計測・予算アサート） | 計測なしに管理なし。マクロTick/会戦フレームの所要を可視化＋退行検知 |

### 推奨着手順
`PERF-1`（計測基盤＝全ての前提）→ `PERF-6`（ティック頻度の監査＝最も安く効く）→ `PERF-3`（キャッシュ＆無効化）→ `PERF-2`（シミュLOD＝根治）→ `PERF-5`（無制限増加の上限）→ `PERF-4`（N²ガードレール）→ `PERF-7`（会戦ヘッドルーム）。

> 本EPICは機能を足さない。**既存の良い規律を計測可能・強制可能にして、今後の追加で終盤ラグを生ませない**ための投資。CLAUDE.md「スケーラビリティ規律」節と対。
